How firmware actually runs
— and what your app does to make it

Two figures. The first traces a chip from the moment power arrives to the moment your code is looping forever. The second traces your C source from the Compile button to the marketplace.

Target ARM Cortex-M4 FLASH 0x08000000 · 512K RAM 0x20000000 · 128K Signals 173

Firmware

Code that lives inside a device instead of on a computer. Your microwave has it. You never installed it.

Bare metal

No operating system underneath. No printf, no files, no internet. Just the chip and your instructions.

The consequence

Nothing sets anything up for you. Even getting to main() is work your own code has to do first.

The boot sequence

Press Power on and step through it. Watch the memory map on the right — every step below is real work performed by build/startup.c, a file checked into your repo that gets compiled into every single build.

The chip is asleep

No power, nothing running. Your compiled code is sitting in FLASH memory, exactly where it was written. FLASH keeps its contents with the power off — that is the whole point of it.

This is why your signals are always zero

Step 4 zeroes .bss. All 173 signals live in .bss. Nothing ever writes them afterwards.

On a real car, an XCP tool would be attached over CAN, streaming live pedal and brake values into those memory slots many times a second. Cortex Studio has no cable, no protocol, no car. So Acc_Pedal_Pos is set to 0 at boot and stays 0 forever.

That is not a bug — it is the honest edge of the scope. The app is answering “is this code written correctly for this ECU’s vocabulary?”, not “what is the pedal doing right now?”

Figure 2

What happens when you click Compile

Nine steps from the button to the marketplace. Step 6 is the one the assignment is really grading.

01
Your browser saves first

Quietly, before anything else. The server compiles the saved version — without this you would fix a typo, hit Compile, and get the error from your old code.

02
Three guards on the door

Is this actually your workspace? Is there any code in it? Do you already have a build running? That last one is a traffic light — one build per person, so nobody can click Compile 500 times and melt the server.

03
A ticket, returned instantly

The server writes down “build #12, waiting” and replies immediately — before compiling anything. Like a restaurant buzzer: you get the buzzer now, the food later. Otherwise the page would freeze for two seconds.

04
A worker picks it up

Two workers exist. Not unlimited — so a hundred people clicking at once forms an orderly queue instead of launching a hundred compilers.

05
Five files into a fresh folder

user.c is yours. signals.h and signals_def.c are generated from the A2L. startup.c and link.ld are fixed files from the repo that you cannot touch.

06
The sealed room

You are running a stranger’s code on your server. So the compiler does not run on your server — it runs in a disposable container with every door locked, and the container is destroyed afterwards.

  • no networkCannot phone home or send anything out.
  • 256 MB capCannot eat all the memory.
  • 64 processes maxCannot spawn copies until the server dies.
  • read-only diskOne temp folder writable. Nothing else.
  • runs as nobodyNo administrator powers inside.
  • 15 second limitHangs, and it is killed outright.
  • fixed scriptThe room knows two commands and ignores anything you send it.
  • fresh every timeNothing survives between builds.
07
Two commands, always the same two

arm-none-eabi-gcc compiles everything into out.elf. Then objcopy strips it down to out.bin — raw instructions, the actual firmware. You never supply a command, a filename, or a setting. Only the contents of one file.

08
A suspicious verdict

Success requires all three: the compiler said OK, and the files actually exist, and they are under 2 MB. If the compiler claims success but produced nothing, it is marked failed. No silently pretending.

09
Onto the shared shelf

A marketplace entry is created, visible and downloadable by anyone logged in. The temp folder is deleted no matter what happened.

meanwhile Your browser asks “done yet?” every 0.8 seconds, and paints the real compiler output — genuine errors and warnings, not a green tick — the moment it arrives.

The whole thing in one breath

A car computer stores values in numbered memory slots. The A2L file labels them. Your app reads that label sheet, turns it into a C vocabulary list, lets you write code using those names, and compiles it into real Cortex-M firmware — inside a sealed, disposable, internet-less container, because the code came from a stranger. The firmware would genuinely boot. It is just never flashed, so the values it reads are the zeros that step 4 put there.
